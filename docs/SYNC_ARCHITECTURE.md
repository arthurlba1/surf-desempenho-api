# Sync Domain - Offline-First Architecture (Fase 1)

## Overview

The `sync` domain enables offline-first operations by:
1. Accepting commands from mobile clients (even when executed offline)
2. Ensuring idempotency (same command never processed twice)
3. Handling conflicts with simple strategies (LWW for low-concurrency scenarios)
4. Providing replay/retry mechanisms

## Key Assumptions (from Product)

- **Low concurrency**: Typically the same user (coach) manages their school's data
- **Rare conflicts**: Different users editing the same resource is uncommon
- **Simple conflict resolution**: Last-Write-Wins (LWW) is acceptable for most cases

## Command Structure

Every command from the app includes:

```typescript
{
  commandId: string;           // UUID v4, generated by client
  commandType: string;          // e.g., 'UPDATE_PROFILE', 'CREATE_SESSION'
  actorUserId: string;          // User executing the command
  schoolId?: string;            // Tenant context (if applicable)
  payload: object;              // Command-specific data
  clientSequence: number;       // Monotonic counter per user+school
  createdAt: string;            // ISO timestamp from client
  version?: number;             // For optimistic locking (optional)
}
```

### Command ID

- **Format**: UUID v4
- **Generated by**: Client (app)
- **Purpose**: Global uniqueness for idempotency

### Client Sequence

- **Format**: Monotonic integer
- **Scope**: Per `(actorUserId, schoolId)` pair
- **Purpose**:
  - Detect missing/out-of-order commands
  - Allow ordered replay on reconnection

## Command Inbox (Backend)

Schema:

```typescript
{
  _id: ObjectId,
  commandId: string (unique index),
  commandType: string,
  actorUserId: string,
  schoolId?: string,
  payload: object,
  clientSequence: number,
  createdAt: Date,
  receivedAt: Date,
  status: 'RECEIVED' | 'PROCESSING' | 'APPLIED' | 'FAILED',
  appliedAt?: Date,
  error?: string,
  result?: object,
  version?: number
}
```

Indexes:
- `{ commandId: 1 }` unique
- `{ actorUserId: 1, schoolId: 1, clientSequence: 1 }`
- `{ status: 1, receivedAt: 1 }`

## Idempotency Strategy

### 1. Command Deduplication

```typescript
// Pseudocode
async function processCommand(cmd: Command): Promise<CommandResult> {
  // Step 1: Upsert command inbox with commandId
  const existing = await CommandInbox.findOneAndUpdate(
    { commandId: cmd.commandId },
    {
      $setOnInsert: {
        ...cmd,
        receivedAt: new Date(),
        status: 'RECEIVED'
      }
    },
    { upsert: true, returnDocument: 'after' }
  );

  // Step 2: If already processed, return cached result
  if (existing.status === 'APPLIED') {
    return { success: true, data: existing.result, cached: true };
  }

  // Step 3: If currently processing by another instance, wait/retry
  if (existing.status === 'PROCESSING') {
    await sleep(100);
    return processCommand(cmd); // Retry
  }

  // Step 4: Mark as processing
  await CommandInbox.updateOne(
    { commandId: cmd.commandId },
    { $set: { status: 'PROCESSING' } }
  );

  try {
    // Step 5: Apply command to domain
    const result = await applyDomainCommand(cmd);

    // Step 6: Mark as applied + cache result
    await CommandInbox.updateOne(
      { commandId: cmd.commandId },
      {
        $set: {
          status: 'APPLIED',
          appliedAt: new Date(),
          result
        }
      }
    );

    return { success: true, data: result };
  } catch (error) {
    // Step 7: Mark as failed
    await CommandInbox.updateOne(
      { commandId: cmd.commandId },
      {
        $set: {
          status: 'FAILED',
          error: error.message
        }
      }
    );
    throw error;
  }
}
```

### 2. Domain-Level Idempotency

Some commands are naturally idempotent (e.g., "set field X to value Y"). For others:

- **Optimistic locking**: Use `version` field
  - Client sends `version` from last known state
  - Backend compares with current DB version
  - Reject if mismatch (conflict)

- **Semantic checks**: E.g., "add member" can check if already member

## Conflict Resolution (Fase 1)

### Strategy: Last-Write-Wins (LWW)

For most operations, **accept the latest command** based on `appliedAt` timestamp.

**When to use LWW:**
- Update profile fields
- Update school settings
- Mark attendance (by coach)

**Example:**
```
Command A (applied 10:00): Set user.birthdate = "1990-01-01"
Command B (applied 10:05): Set user.birthdate = "1991-01-01"
Result: user.birthdate = "1991-01-01" (last write wins)
```

### Strategy: Reject with Conflict Error

For operations where LWW isn't safe:
- Delete operations
- Membership status changes (approve/block)
- Payment records

**Conflict response:**
```json
{
  "statusCode": 409,
  "message": "Conflict: resource was modified. Please refresh and try again.",
  "conflict": {
    "commandId": "abc-123",
    "field": "status",
    "clientVersion": 5,
    "serverVersion": 7
  }
}
```

Client must:
1. Fetch latest data (`GET /auth/me` or domain endpoint)
2. Merge local changes (if possible)
3. Retry with new `version`

## Client-Side Queue Management

### Queue Schema (SQLite)

```sql
CREATE TABLE command_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  command_id TEXT UNIQUE NOT NULL,
  command_type TEXT NOT NULL,
  actor_user_id TEXT NOT NULL,
  school_id TEXT,
  payload TEXT NOT NULL, -- JSON
  client_sequence INTEGER NOT NULL,
  created_at TEXT NOT NULL, -- ISO timestamp
  version INTEGER,
  status TEXT DEFAULT 'PENDING', -- PENDING | SENT | APPLIED | FAILED
  retry_count INTEGER DEFAULT 0,
  last_error TEXT,
  applied_at TEXT
);

CREATE INDEX idx_status ON command_queue(status);
CREATE INDEX idx_sequence ON command_queue(actor_user_id, school_id, client_sequence);
```

### Queue Operations

#### Enqueue

```typescript
async function enqueueCommand(cmd: Command): Promise<void> {
  await db.insert('command_queue', {
    command_id: cmd.commandId,
    command_type: cmd.commandType,
    actor_user_id: cmd.actorUserId,
    school_id: cmd.schoolId,
    payload: JSON.stringify(cmd.payload),
    client_sequence: cmd.clientSequence,
    created_at: cmd.createdAt,
    version: cmd.version,
    status: 'PENDING'
  });
}
```

#### Sync Loop

```typescript
async function syncPendingCommands(): Promise<void> {
  const pending = await db.query(
    'SELECT * FROM command_queue WHERE status IN (?, ?) ORDER BY client_sequence ASC',
    ['PENDING', 'FAILED']
  );

  if (pending.length === 0) return;

  // Batch commands
  const batch = pending.slice(0, 10); // Max 10 per batch

  try {
    const response = await apiClient.post('/sync/commands', {
      commands: batch.map(cmd => ({
        commandId: cmd.command_id,
        commandType: cmd.command_type,
        actorUserId: cmd.actor_user_id,
        schoolId: cmd.school_id,
        payload: JSON.parse(cmd.payload),
        clientSequence: cmd.client_sequence,
        createdAt: cmd.created_at,
        version: cmd.version
      }))
    });

    // Mark processed as APPLIED
    for (const commandId of response.data.processed) {
      await db.update(
        'command_queue',
        { status: 'APPLIED', applied_at: new Date().toISOString() },
        { command_id: commandId }
      );
    }

    // Mark failed
    for (const failure of response.data.failed) {
      await db.update(
        'command_queue',
        { 
          status: 'FAILED', 
          last_error: failure.error,
          retry_count: db.raw('retry_count + 1')
        },
        { command_id: failure.commandId }
      );
    }

  } catch (error) {
    // Network error - will retry later
    console.error('Sync failed:', error);
  }
}
```

#### Retry Strategy

- **Auto-retry**: Every 30s when online
- **Exponential backoff**: For failed commands (1min, 2min, 4min, ...)
- **Max retries**: 5 attempts, then mark as "needs manual review"
